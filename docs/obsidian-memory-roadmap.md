# Obsidian Memory Roadmap

This is the working plan for making Obsidian the local memory and connection layer for Finance Superbrain.

The priority is to build a dependable research and decision memory system before returning to public production health work. Hosted production still matters, but the current product direction is to harden the backend, frontend, and data model around trustworthy financial workflows before rushing public pilot deployment.

## Product Standard

Finance Superbrain is a financial application. The system must be designed as if mistakes can affect real capital decisions.

Required engineering posture:

- backend state must be auditable, deterministic where possible, and tested
- frontend workflows must make decision state, provenance, confidence, and risk visible
- money-adjacent actions must prefer explicit review steps over hidden automation
- PostgreSQL remains the production source of truth
- Next.js remains the web application shell unless there is a concrete reason to split surfaces
- React Native can become a later mobile client, but should use the same API contracts and backend audit model
- Obsidian is a local memory layer, not the canonical transaction or portfolio state store

## Target Vault Shape

Recommended vault layout:

```text
Finance Superbrain/
  Activity/
  Connections/
  Decisions/
  Human Inbox/
  Indexes/
  Investigations/
  Lessons/
  Portfolio/
  Project/
  Reviews/
```

Managed directories are generated by Finance Superbrain and should not be hand-edited as source material. Human-authored notes should go into `Human Inbox/` first.

## Human Memory Note Format

Use this shape for notes that should become retrieval memory:

```markdown
---
fs_import: true
title: CPI breadth and bond reaction pattern
lesson_type: reinforcement
themes: [inflation, rates, breadth]
assets: [TLT, DXY, SPY]
tags: [human-memory, cross-asset]
---

# CPI breadth and bond reaction pattern

When CPI is hot but long bonds stop selling off, check whether the market is shifting from inflation fear to growth fear.
```

Import flow:

1. Write the note under `Finance Superbrain/Human Inbox`.
2. Run `npm run ops:obsidian-import`.
3. Review the dry-run candidates.
4. Run `npm run ops:obsidian-import -- --apply`.
5. Open the Library page and confirm the imported memory appears.

## Connector Build Order

### 1. Canonical Memory Runbook

Goal: make the existing export/import bridge easy to operate repeatedly.

Deliverables:

- document environment variables and vault setup
- add example Human Inbox notes
- add a local acceptance checklist
- keep the generated subtree and human inbox rules explicit
- add a durable automatic sync command for project work sessions
- add an Obsidian plugin that refreshes a visible status note and review snapshot

Acceptance:

- a user can create a clean vault, export system memory, import a human note, and find that memory in the app without reading code
- a user can keep `Project/Work Session.md` current by running the watch command once
- a user can open Obsidian and see the latest sync/review status without opening the web app

### 2. Phase And Decision Traceability

Goal: make long-running project memory visible in Obsidian.

Deliverables:

- phase ledger notes exported or generated into Obsidian
- decision records for major architecture choices
- links from phase notes to investigations, tests, scripts, and known risks

Acceptance:

- Obsidian can answer what each phase shipped, how it is validated, and what risk remains
- current baseline: generated `Project/Work Session.md`, `Project/Phase Ledger.md`, architecture decision records, `Project/Build Log.md`, `Project/Risk Register.md`, `Project/Validation History.md`, and `Project/Data Inventory.md`
- `Project/Phase Ledger.md` is sourced from `docs/phase-ledger.md`
- explicit per-phase evidence links now cover repo refs, scripts, tests, risks, and deployment status parsed from `docs/phase-ledger.md`
- architecture decision records now link major product boundaries back to relevant phases and validation evidence
- plugin-generated sync notes and app-generated work-session notes now point at the same sync/review state while staying out of `Human Inbox/`

### 3. Connection Finder

Goal: surface non-obvious relationships across imported human memory, lessons, investigations, decisions, and portfolio candidates.

Initial signals:

- repeated assets
- repeated themes
- matching macro regimes
- similar failed assumptions
- recurring confidence labels
- lessons that contradict current decisions
- stale portfolio candidates tied to active investigations

Deliverables:

- backend connection report builder for repeated asset and theme signals
- exported `Connections/` notes
- generated `Connections Index.md`
- optional web surface in Library or workspace

Acceptance:

- the system can produce a short list of explainable connections with source links and reason codes
- the web review surface is now available in the Library and remains read-only

### 4. Review Queue Before Import Apply

Goal: prevent accidental memory pollution.

Deliverables:

- API route for import candidates
- web review UI for accept/reject before apply
- audit record for imported, skipped, and rejected notes

Acceptance:

- no human note becomes retrieval memory without an explicit reviewed action in the app
- the review queue exposes importable notes, and selected notes are imported only when explicitly approved
- reject-all review decisions are recorded without importing retrieval memory
- the Obsidian plugin reflects the same import-review snapshot inside the vault
- status: implemented locally, with API regression coverage for selected-only import and reject-all audit logging

### 5. Rich Linking

Goal: connect imported notes to the exact objects they support.

Deliverables:

- support `linked_decision_brief_id`
- support `linked_portfolio_candidate_id`
- support `linked_investigation_id`
- render backlinks in export notes
- show linked memory in app object pages

Acceptance:

- a decision brief can show which Obsidian memories influenced it, and an Obsidian note can link back to the app object
- imported notes can surface investigation, decision, and portfolio links in both the app and exported notes
- status: implemented locally for imported lesson metadata, generated Obsidian backlinks, and Library linked-object chips

### 6. Mobile Client Later

Goal: add mobile capture after the backend memory model is stable.

Recommended path:

- keep Next.js as the primary desk workflow
- add React Native only for capture, review, alerts, and lightweight portfolio checks
- use the same PostgreSQL-backed API and audit model

Non-goal for now:

- no mobile-first rewrite
- no direct mobile writes to Obsidian
- no transaction execution from an unreviewed memory workflow

## Backend Safety Rules

Obsidian integration must preserve these boundaries:

- PostgreSQL is source of truth for app state
- Obsidian import creates retrieval memory only
- generated export files remain managed and reproducible
- broad vault import stays blocked
- duplicate imports are prevented by content hash
- every imported note keeps source path and hash metadata
- any future bidirectional sync requires explicit conflict handling and audit records

## Frontend Safety Rules

The frontend should make memory provenance obvious:

- distinguish generated system memory from human-imported memory
- show source path, import status, and linked objects
- make confidence and uncertainty visible
- require review before memory changes influence money-adjacent workflows
- avoid hiding important risk behind polished summaries
